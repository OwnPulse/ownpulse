# ADR-0012: Data Privacy Hardening

**Date:** 2026-03-21
**Status:** Accepted
**Deciders:** Tony Ferrell

---

## Context

OwnPulse stores sensitive health data — lab results, genetic records, substance use, subjective wellbeing scores. The platform's core promise is data sovereignty: users control their own data. Before accepting real users, we needed to close security gaps that could undermine trust.

The starting posture was strong on fundamentals (TLS, Tailscale VPN, JWT auth, rate limiting, httpOnly cookies, SOPS+SealedSecrets) but had gaps: health data stored as plaintext in Postgres, no network isolation between pods, no audit trail, no key rotation, no security headers, and default secrets accepted silently in production.

**Constraint:** Solo dev, 4GB droplet, self-hosted target audience. Solutions must be simple, low-overhead, and work for people running their own instance.

---

## Decision

Implement a three-tier security hardening plan. Tiers 1 and 2 are implemented. Tier 3 is deferred.

### Tier 1 — Must-have for launch (implemented)

| Protection | How |
|-----------|-----|
| **Reject default secrets** | Server panics on startup if JWT_SECRET or ENCRYPTION_KEY are defaults and WEB_ORIGIN is non-localhost |
| **Security headers** | HSTS, X-Content-Type-Options, X-Frame-Options on all ingress; CSP on web |
| **NetworkPolicies** | kube-router deployed alongside Flannel for policy enforcement; Postgres only reachable from API pods |
| **Encrypted backups** | pg_dump piped through age encryption (dual recipient keys) before upload to DO Spaces |
| **Security documentation** | `docs/security.md` — full security model for users and self-hosters |

### Tier 2 — Defense in depth (implemented)

| Protection | How |
|-----------|-----|
| **Postgres TLS** | Self-signed cert generated by init container; API connects with `sslmode=require` |
| **Non-superuser DB role** | `ownpulse_app` role with only DML grants; superuser reserved for migrations |
| **Row-Level Security** | RLS policies on all 14 user-data tables; enforced when connecting as non-superuser |
| **Audit logging** | `data_access_log` table tracking exports, deletes, and account operations; async insert via tokio::spawn |
| **Encryption key rotation** | Versioned ciphertext (`v1:...`); `ENCRYPTION_KEY_PREVIOUS` for fallback during rotation |

### Tier 3 — Deferred

| Protection | Why deferred |
|-----------|-------------|
| **Column-level encryption** | Would block server-side analytics (correlations, aggregates) planned for Phase 3. Acceptable because DO encrypts volumes at rest and backups are age-encrypted. |
| **Encrypted exports** | Nice-to-have; HTTPS protects in transit, user's device is the trust boundary |
| **Backup verification** | Weekly restore-test CronJob; low priority until backup volume grows |

## Alternatives Considered

### Full-disk encryption with user-held keys

Encrypt the entire Postgres data directory with LUKS, user provides the key at boot. Maximum security but terrible UX for self-hosters (manual unlock on every restart) and incompatible with managed hosting.

### Column-level encryption for all fields

Encrypt every sensitive column with AES-256-GCM at the application layer. Considered and initially planned, but rejected because it prevents SQL-level aggregation needed for the correlation engine (`stats.rs`). Server-side queries like `AVG(value)`, `PERCENTILE_CONT`, and cross-user cooperative aggregates become impossible without decrypting all rows into memory first.

**Compromise:** Integration tokens (OAuth credentials) are already AES-256-GCM encrypted. Health data relies on volume encryption + RLS + network isolation. Column encryption remains an option for the highest-sensitivity fields (genetic data) if needed.

### Service mesh (Linkerd/Istio) for mTLS

Would encrypt all pod-to-pod traffic with mutual TLS. Rejected as overkill for a single-node cluster with 3 services. NetworkPolicies + Postgres TLS achieve the same practical security at a fraction of the resource cost (~30MB for kube-router vs ~200MB+ for Linkerd).

### Vault for secrets management

HashiCorp Vault for dynamic secrets, automatic rotation, and audit logging. Rejected as overengineered for a solo-dev MVP. SOPS + SealedSecrets + manual rotation with clear runbooks is sufficient. Vault adds ~500MB RAM and significant operational complexity.

## Consequences

**Positive:**
- Self-hosters get secure defaults out of the box (server refuses to start with dev secrets)
- Defense-in-depth: even if application code has a bug, RLS prevents cross-user data leakage
- Encrypted backups mean a Spaces bucket leak doesn't expose health data
- Key rotation is non-disruptive (set new key, old values still decrypt via fallback)
- Security model is documented and auditable

**Negative / tradeoffs:**
- kube-router adds ~30MB RAM to the already-tight 4GB droplet
- RLS adds a filter to every query plan (negligible overhead but non-zero)
- Audit logging adds database writes on sensitive operations (async, non-blocking)
- Column-level encryption is deferred — health data is plaintext at the Postgres layer (protected by volume encryption and access controls, not application-level crypto)

**Risks:**
- RLS depends on connecting as `ownpulse_app` (non-superuser). If DATABASE_URL accidentally uses the `postgres` superuser, RLS is silently bypassed. Mitigated by the Helm chart's initdb script creating the role automatically.
- kube-router requires Flannel-compatible mode. If the CNI is changed (e.g., to Cilium), kube-router should be removed to avoid conflicts.
- Backup CronJob needs DO Spaces credentials (sealed). If the SealedSecret is missing, backups silently fail. Consider adding a monitoring alert for backup age.
